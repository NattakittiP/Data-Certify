DATA-CERTIFY -- maximum_curvature_mc() bugfix: full corpus recalibration report
TENTH CALIBRATION PASS, 2026-07-16
=================================================================================

SOURCE OF THIS PASS
--------------------
A real floating-point bin-edge bug in data_certify/stats.py::maximum_curvature_mc()
was found during a user-requested independent 3-round re-verification of Group D1(d)
(Docs/03_Paper_Prep/DATA-CERTIFY_Cross_Agency_Catalog_Merge_Case_Study.md, Section 1a). The bug: the old
implementation built histogram bin edges with np.arange(lo, hi+bin_width, bin_width)
and binned via np.histogram; np.arange's accumulated floating-point drift could push a
bin edge conceptually meant to sit exactly at a reported (0.1-precision) magnitude a few
ULPs off, silently misclassifying every event at that value into the neighboring bin.

Fix (final, shipped): bin_idx = floor((m - lo) / bin_width + 1e-9); preserves the
original left-edge binning convention (so continuously-valued catalogs like `nz`, which
the old code was NOT actually wrong on, are unaffected) while eliminating the float-drift
misclassification for quantized data. A first candidate fix (nearest-bin rounding) was
tested and rejected because it silently shifted the binning convention for `nz`
specifically (Mc 1.838 -> 1.938) despite the original code not being wrong there -- see
data_certify/stats.py's own module comment for the full derivation.

CORPUS-WIDE Mc IMPACT (before any A2 propagation)
---------------------------------------------------
175 of 985 total corpus datasets (calibration corpus + adversarial holdout) had a
different computed Mc after the fix. Of the 968-dataset main calibration corpus
specifically, only 18 datasets had a resulting A2 (Gutenberg-Richter b-value
plausibility) score change of a nonzero amount once propagated through
gr_b_value_aki() -- the remaining Mc shifts had no material effect on the derived
b-value/score. A 19th affected dataset, `emsc_raw`, belongs to the D1(d) case study's
own live-fetched data and is NOT part of the 968-dataset calibration corpus (see
DATA-CERTIFY_Cross_Agency_Catalog_Merge_Case_Study.md Section 1a for its own before/after numbers).

CONCLUSION
----------
The maximum_curvature_mc() fix's corpus-wide impact, once fully propagated through
score_matrix.csv regeneration, EWM weight recalibration, and threshold re-validation,
is CONFIRMED to be below the precision most of this project's documentation quotes
weights to (4 decimal places) and produces ZERO ADMIT/CONDITIONAL/REJECT decision
changes for any of the 968 corpus datasets. The open item raised in D1(d)'s Section 1a
("whether any of the other 19 affected datasets... require updates to previously
reported Group A-C calibration/validation numbers") is RESOLVED: no previously reported
Groups A-C headline number (axis weights, within-axis weights, decision thresholds,
false-admit/false-reject counts) requires a substantive correction beyond the routine
weight-value updates already applied to data_certify/_constants.py (deltas all <0.004,
i.e. below the 4-decimal-place precision those docs already quote). Individual dataset
A2 sub-scores did shift (up to 0.79 for one dataset), but this had no effect on any
aggregate/reported statistic beyond the weight table itself.
